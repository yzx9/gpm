## Problem Statement

The Rust backend library (`rustpass/`) uses synchronous `std::fs` for all file I/O. On Android devices with slow flash storage, this causes UI freezes during normal operations — listing entries, decrypting passwords, pulling from git. The library has zero async dependencies despite being consumed by a Tauri v2 app that runs on tokio.

## What Makes This Cool

Two things make this more than a mechanical `s/std::fs/tokio::fs/` refactor:

1. **Cancellable operations wired to Android lifecycle.** Every public `Store` method takes a `CancellationToken`. The Tauri layer creates one per invocation. The Android lifecycle plugin cancels all in-flight tokens when the activity pauses. This is structured cancellation semantically tied to the mobile lifecycle — not just "don't block the executor." If the user navigates away during a `WalkDir` traversal or mid-decrypt, the operation dies cleanly instead of burning I/O for nobody.

2. **rustpass as a standalone async Rust library.** No Rust age library ships with an async store API. This crate becomes a reusable `async fn` surface that other Tauri apps, CLI tools, or async Rust projects can consume directly without `spawn_blocking` wrapping.

## Constraints

- **git2 stays synchronous.** It's a C library binding. `clone_repo`, `pull_repo`, and all git operations remain sync internally, wrapped in `spawn_blocking` at the `Store` level.
- **Age decryption stays synchronous.** The `age` crate does streaming decryption through `Read` traits — it's CPU-bound, not I/O-bound. Only the file read (loading encrypted bytes) goes async. The decryption itself is sync computation on in-memory data.
- **Sequential operations don't become concurrent.** A single decrypt flow is: load config → load identity → read file → decrypt → parse. These steps depend on each other. Async's value here is not concurrency within one operation — it's **non-blocking** execution and **cancellation**.
- **`tokio::fs` is `spawn_blocking` under the hood.** Accepted premise. On Linux/macOS/Android, there is no true async file I/O. The function signatures change, but runtime behavior is thread-pool offload either way.

## Premises

1. **`tokio::fs` is functionally equivalent to `spawn_blocking(|| std::fs::...)`.** Operating systems do not provide true async file I/O on target platforms. `tokio::fs` internally uses a blocking thread pool. Accepted — the refactor is for API shape, not runtime behavior.
2. **The runtime behavior will be identical to spawn_blocking wrappers.** Sequential operations (config → identity → decrypt → copy) gain no concurrency benefit from async. Accepted — the value is the clean async API surface for library consumers.
3. **git2 cannot be made async.** It's a C library binding. Accepted — git operations use `spawn_blocking` at the `Store` level regardless.
4. **Cancellation tokens are the architectural payoff.** Without them, this refactor is mechanical substitution. With them, rustpass gains a genuinely useful capability that `spawn_blocking` alone cannot provide.
5. **Test migration happens in the same commit.** Every `#[test]` becomes `#[tokio::test]`. Mechanical but must not be deferred — broken tests block the merge.

## Cross-Model Perspective

Second opinion (Claude subagent) raised three key points:

1. **Cancellable operations with `CancellationToken`** mapped to Android lifecycle events (`Activity.onPause`/`onStop`). This transforms the refactor from mechanical to architecturally interesting. The cancellation token is a library-level concern — checked at `WalkDir` iterator level for `list_entries`, between file read and age decryption for `decrypt_file`, after network fetch for `pull_repo`.

2. **rustpass as a principled standalone library.** The user cares about the crate having a clean API that someone could pick up and use correctly without knowing it was originally built for an Android password manager. This is an aesthetic judgment about API design, not a performance argument.

3. **No existing Rust age library ships with an async store API.** rustpass would be novel in this space. Reference: `gopass` (Go) has the same `List/Get/Sync` API philosophy, but no Rust equivalent exists.

## Approaches Considered

### Approach A: spawn_blocking at Tauri boundary (minimal viable)

Wrap every Tauri command in `tokio::task::spawn_blocking`. Zero changes to rustpass internals.

**Effort:** S (human: ~30 min / CC: ~10 min) | **Risk:** Low | **Completeness:** 6/10

- ✅ Solves the UI freeze immediately with zero risk
- ✅ Tests pass unchanged, rustpass stays simple
- ❌ rustpass stays a sync library — consumers must always wrap in `spawn_blocking`
- ❌ Every new Tauri command needs the wrapper — easy to forget

### Approach B: Hybrid — spawn_blocking + targeted async (recommended by office hours)

Keep rustpass sync internally, add `spawn_blocking` at the boundary AND targeted async for parallel operations.

**Effort:** S (human: ~2 hours / CC: ~30 min) | **Risk:** Low | **Completeness:** 8/10

- ✅ Fixes the freeze AND targets where async genuinely helps
- ✅ Keeps sync core, adds async only where it matters
- ❌ Two patterns coexist — sync core + async boundary
- ❌ Not a clean async library — but per premises, that's not a real loss

### Approach C: Full async crate (CHOSEN)

Rewrite rustpass with `async fn` throughout using `tokio::fs`. Every function signature changes. git2 operations use `spawn_blocking` at the `Store` level. Cancellation tokens added as a second phase.

**Effort:** L (human: ~2-3 days / CC: ~4-6 hours) | **Risk:** Medium | **Completeness:** 10/10

- ✅ Clean async API surface — other async Rust projects consume rustpass directly
- ✅ Cancellation tokens give genuine architectural value beyond spawn_blocking
- ✅ Every layer is consistent — no mixed sync/async for contributors
- ❌ Every function signature changes. Every call site changes. Every test changes.
- ❌ git2 operations still need `spawn_blocking` — can't escape it for network I/O

## Recommended Approach

**Approach C: Full async crate, delivered in two phases.**

Phase 1: Mechanical async migration (weekend). Replace `std::fs` with `tokio::fs`, convert all public methods to `async fn`, migrate tests to `#[tokio::test]`. No cancellation tokens yet — get the basic shape right first.

Phase 2: Cancellation tokens (following week). Add `CancellationToken` to public `Store` methods, wire to Android lifecycle via Tauri events. This is the architectural payoff that makes the refactor worth more than spawn_blocking.

### Migration Order

**Step 1: `rustpass/Cargo.toml`**
Add tokio and tokio_util as dependencies:

```toml
[dependencies]
tokio = { version = "1", features = ["fs"] }
tokio_util = { version = "0.7", features = ["rt"] }  # for CancellationToken (Phase 2)

[dev-dependencies]
tokio = { version = "1", features = ["rt", "macros"] }
```

Note: dev-dependencies need `"rt"` + `"macros"` for `#[tokio::test]` to work. The main dependency needs `"fs"` for `tokio::fs::*`. Features are additive — both sets are available in tests.

**Step 2: `rustpass/src/config.rs`** (cold path, simplest migration)
Every `std::fs` call → `tokio::fs`. Key functions:

- `save_identity_raw` → `async fn save_identity_raw`
- `save_identity` → `async fn save_identity`
- `load_identity` → `async fn load_identity`
- `delete_identity` → `async fn delete_identity`
- `save_repo_config` → `async fn save_repo_config`
- `load_repo_config` → `async fn load_repo_config`
- `clear_all` → `async fn clear_all`
- `is_configured`, `repo_config_exists` — **stay sync** (`.exists()` checks, effectively free)
- `is_identity_encrypted` — **becomes async** (calls `load_identity` which is now async; this is a transitive dependency the implementer must catch)

**Step 3: `rustpass/src/recipient.rs`**
One function touches the filesystem: `list_recipients` → `async fn list_recipients`. The test module (`list_recipients_from_gopass_file`, etc.) must also become `#[tokio::test]` — not just "minor" test fixture changes.

**Step 4: `rustpass/src/crypto.rs`**
`decrypt_file` → `async fn decrypt_file` (only the file read goes async):

```rust
pub async fn decrypt_file(file_path: &Path, identity_bytes: &[u8]) -> Result<Vec<u8>, Error> {
    let encrypted = tokio::fs::read(file_path).await.map_err(...)?;
    decrypt_bytes(&encrypted, identity_bytes)  // sync, CPU-bound — runs on executor thread
}
```

`decrypt_bytes`, `encrypt_identity`, `decrypt_identity` — **stay sync** (pure computation on in-memory data).

**Decision: does `decrypt_bytes` need `spawn_blocking`?** Age decryption is CPU-bound but typically completes in <10ms for single entries. For the password manager use case, this is fast enough to run on the tokio executor thread without starving other tasks. If profiling shows it takes >1ms consistently, wrap it: `tokio::task::spawn_blocking(move || decrypt_bytes(&encrypted, &identity)).await?`. Recommendation: skip `spawn_blocking` for `decrypt_bytes` in Phase 1. Add it in Phase 2 if profiling warrants it. The CPU-bound scrypt in `decrypt_identity` (called by `unlock`) DOES get `spawn_blocking` because scrypt is intentionally slow (~100ms+).

**Step 5: `rustpass/src/store.rs`** (the facade)
Every method that calls an async `Config`/`recipient`/`crypto` method becomes async:

- `list` → `async fn list` (hot path)
- `get` → `async fn get` (hot path — calls `load_repo_config`, `load_identity`, `decrypt_file`)
- `sync` → `async fn sync` (hot path — config load async, git stays sync via `spawn_blocking`)
- `unlock` → `async fn unlock` (calls `load_identity` now async)
- `clone_only` → `async fn clone_only` (cold path)
- `configure` → `async fn configure` (cold path)
- `reset` → `async fn reset` (cold path)
- `list_entries`, `resolve_entry_path` — **stay sync** (pure CPU / stat-only operations)

**Key: mixed async/sync pattern in `Store` methods.** Several methods call both async fs operations AND sync git/crypto operations. The pattern:

```rust
// Store::sync — async config load + spawn_blocking for git
pub async fn sync(&self) -> Result<PullResult, Error> {
    let config = self.config.load_repo_config().await?;  // async
    let repo_path = config.repo_path.clone();
    let auth = config.git_auth.clone();
    tokio::task::spawn_blocking(move || git::pull_repo(&repo_path, &auth)).await??  // sync git via thread pool
}

// Store::unlock — async config load + spawn_blocking for CPU-bound scrypt
pub async fn unlock(&self, passphrase: String) -> Result<(), Error> {
    let encrypted = self.config.load_identity().await?;  // async
    tokio::task::spawn_blocking(move || {
        crypto::decrypt_identity(&encrypted, &passphrase)  // sync CPU-bound scrypt
    }).await?
}

// Store::clone_only — async cleanup + spawn_blocking for git clone
pub async fn clone_only(&self, url: String) -> Result<PathBuf, Error> {
    let config = self.config.load_repo_config().await?;
    let repo_path = config.repo_path.clone();
    if repo_path.exists() {
        tokio::fs::remove_dir_all(&repo_path).await?;
    }
    tokio::task::spawn_blocking(move || git::clone_repo(&url, &repo_path)).await?
}

// Store::reset — async file cleanup
pub async fn reset(&self) -> Result<(), Error> {
    let config = self.config.load_repo_config().await?;
    let repo_path = config.repo_path.clone();
    if repo_path.exists() {
        tokio::fs::remove_dir_all(&repo_path).await?;  // async dir removal
    }
    self.config.clear_all().await?;
    Ok(())
}
```

**RwLock guard invariant:** `Store` uses `std::sync::RwLock<Option<CachedIdentity>>` for `cached_identity`. Guards are always dropped before any `.await` point in the current code. This invariant MUST be maintained — a guard held across `.await` causes `!Send` compilation errors. If future changes need identity across `.await`, switch to `tokio::sync::RwLock`.

**Step 6: `rustpass/src/git.rs`**
**No changes.** `clone_repo` and `pull_repo` remain sync. The `Store` layer wraps calls in `spawn_blocking` when needed.

**Step 7: `src-tauri/src/lib.rs`** (Tauri binding layer)
Every Tauri command that calls an async `Store` method becomes `async fn` (Tauri v2 supports this natively):

- `list_entries` → `async fn list_entries`
- `pull_repo` → `async fn pull_repo`
- `copy_password` → already async, `state.store.get().await` now
- `show_password` → `async fn show_password`
- `unlock` → already uses `spawn_blocking`, simplify to direct `.await`

**Step 8: Tests**
Every `#[test]` → `#[tokio::test]`. Test setup code that uses `std::fs::write` directly (e.g., `config.rs` test `repo_config_backward_compat_no_ssh_fields`) stays sync — there is no need to convert test setup to `tokio::fs`. Only the test functions themselves become async.

### Files NOT Changed

- `rustpass/src/git.rs` — C library binding, stays sync
- `rustpass/src/ssh.rs` — pure computation, zero I/O
- `rustpass/src/entry.rs` — data type only
- `rustpass/src/secret.rs` — data type + parsing
- `rustpass/src/error.rs` — error types only
- `rustpass/src/identity.rs` — may need minor async if it calls config methods
- `rustpass/src/recipient.rs` tests — test functions become `#[tokio::test]`, test setup stays sync

### Phase 2: Cancellation Tokens (design sketch)

```rust
// In store.rs
impl Store {
    pub async fn list(&self, cancel: CancellationToken) -> Result<Vec<Entry>, Error> {
        let config = self.config.load_repo_config().await?;
        if cancel.is_cancelled() { return Err(Error::Cancelled); }
        let entries = list_entries(&config.repo_path)?;
        Ok(entries)
    }

    pub async fn get(&self, entry_path: &str, cancel: CancellationToken) -> Result<Secret, Error> {
        let config = self.config.load_repo_config().await?;
        if cancel.is_cancelled() { return Err(Error::Cancelled); }
        let identity = self.config.load_identity().await?;
        if cancel.is_cancelled() { return Err(Error::Cancelled); }
        let full_path = resolve_entry_path(&config.repo_path, entry_path)?;
        let encrypted = tokio::fs::read(&full_path).await?;
        if cancel.is_cancelled() { return Err(Error::Cancelled); }
        let decrypted = decrypt_bytes(&encrypted, &identity)?;
        Secret::parse(&decrypted)
    }
}
```

Tauri command layer — `AppState` gains an `active_tokens` field:

```rust
struct AppState {
    store: Store,
    active_tokens: tokio::sync::Mutex<Vec<CancellationToken>>,  // async mutex for .await compatibility
    // ... existing fields ...
}

#[tauri::command]
async fn list_entries(state: State<'_, AppState>) -> Result<Vec<Entry>, AppError> {
    let cancel = CancellationToken::new();
    state.active_tokens.lock().await.push(cancel.clone());
    let result = state.store.list(cancel).await;
    // cleanup: remove token from active list (use retain to drop by identity)
    state.active_tokens.lock().await.retain(|t| !t.is_cancelled());
    result.map_err(Into::into)
}
```

Android lifecycle integration: when `Activity.onPause` fires, the Kotlin side emits a Tauri event. The Rust side listens and cancels all active tokens:

```rust
// In app setup (src-tauri/src/lib.rs)
app.listen("android-lifecycle-pause", |_| {
    // cancel all active tokens — this is Phase 2, actual wiring TBD
});
```

Phase 2 scope includes: Kotlin-side lifecycle event emission (minor change to `MainActivity.kt`), Rust-side event listener, and the `active_tokens` field. This is ~1-2 hours of additional work beyond the Phase 1 mechanical migration.

## Open Questions

1. **tokio_util vs hand-rolled cancellation.** `tokio_util::sync::CancellationToken` adds a dependency. An `Arc<AtomicBool>` is zero-dependency but less ergonomic. Decision: start with `tokio_util` — it's well-tested and the dependency is minimal.
2. **spawn_blocking for git in Store vs Tauri layer.** Should `Store::sync()` call `spawn_blocking(|| git::pull_repo())` internally, or should the Tauri command do it? Recommendation: `Store` layer handles it — keeps the Tauri commands thin.
3. **Error type for cancellation.** Add `Cancelled` variant to `rustpass/src/error.rs` in the `ErrorCode` enum. The Tauri layer (`AppError`) should silently handle it — no error toast when the user navigated away. Add it as a non-user-facing variant that maps to a no-op on the frontend.

## Success Criteria

### Phase 1 (mechanical async migration)

- No UI freezes during any file I/O operation on Android
- Every public `Store` method is `async fn`
- Every public `Config` method that touches disk is `async fn`
- All tests pass with `#[tokio::test]`
- git2 operations wrapped in `spawn_blocking` at the `Store` level
- CPU-bound crypto: `decrypt_identity` (scrypt, ~100ms+) wrapped in `spawn_blocking`; `decrypt_bytes` (<10ms) runs on executor thread — profile and add `spawn_blocking` only if needed
- `crypto.rs` decryption stays sync internally — only file reads are async
- Files with zero I/O (`ssh.rs`, `entry.rs`, `secret.rs`) are unchanged
- `error.rs` gains `Cancelled` variant (used by Phase 2)
- `rustpass` crate compiles and passes tests independently of `src-tauri`
- `Store` struct remains `Send + Sync` (guard lifetime invariant documented)

### Phase 2 (cancellation tokens)

- Every public `Store` method accepts `CancellationToken`
- `AppState` has `active_tokens: tokio::sync::Mutex<Vec<CancellationToken>>`
- Android `Activity.onPause` cancels all in-flight tokens
- `Cancelled` error silently handled — no user-facing toast
