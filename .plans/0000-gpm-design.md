## Problem Statement

There is no Android GUI client that can read age-encrypted gopass/password-store repositories. The Android Password Store app is unmaintained and GPG-only. gopass itself is Go/CLI-only. People resort to running gopass inside Termux on Android. The intersection of age encryption + gopass store format + Android is genuinely empty.

This project fills that gap: a read-only, age-only, gopass-compatible password client for Android (and desktop as a bonus), built on Tauri v2 + Rust + Vue 3.

## What Makes This Cool

The trust story IS the product. The "whoa" moment is not speed or features — it's "I can audit exactly what this app does and verify nothing leaves my device." Age-only, no cloud, no analytics, no GPG, no Autofill. The minimal attack surface is the point.

**Second opinion insight:** After every `git pull`, highlight which entries changed, which are new, and which were removed — with commit metadata. This transforms the app from a passive vault into an active trust boundary monitor. Costs almost nothing (git2 already provides diff/log). **Deferred to post-MVP** (see Future section).

## Constraints

- **Read-only MVP:** Clone, pull (ff-only), list, search, decrypt, copy. No edit, push, create, delete, or Autofill.
- **Age-only:** No GPG support. No `.gpg` file handling. No gopass mounts.
- **No cloud sync:** Git is the only sync mechanism. No third-party servers.
- **No analytics or crash logs:** Nothing leaves the device.
- **Android-first via Tauri v2:** Mobile target from day one, desktop as bonus.
- **Security boundaries:** No identity exposure to Vue layer. No secrets in logs/toasts/errors. Clipboard auto-clear after 30 seconds. Primary operation (copy password) avoids IPC entirely — password never reaches WebView. Secondary operation (show password) crosses IPC with strict lifecycle management. Android FLAG_SECURE deferred but planned. Full trust boundary analysis in Security Architecture section.
- **Re-configuration:** MVP supports one repo + one identity. Changing either requires re-running the Setup flow (clears existing local repo). No settings page. No multi-repo support. Explicitly out of scope for MVP.
- **Git authentication:** HTTPS-only with Personal Access Token (PAT) for MVP. PAT stored alongside identity in local config. SSH support is post-MVP (requires libssh2 cross-compilation for Android, significant NDK complexity).
- **Identity model:** MVP supports a single age identity. Multiple identities are post-MVP. The configured identity is tried against every `.age` file — no `.age-recipients` parsing needed for single-identity use cases.
- **Search:** Case-insensitive substring match on entry name and path. Frontend-only, no Rust round-trip.

## Premises

1. **Tauri v2 mobile is mature enough for a read-only password viewer.** No Autofill, no complex native UI, no editing — this avoids the areas where Tauri mobile is roughest. Acceptable risk for MVP.

2. **Rust `age` crate (not kage) is the right crypto choice.** One implementation across desktop and Android. Reference-quality Rust implementation from `str4d/rage`. No Kotlin crypto to audit.

3. **gopass age-store format is stable enough to build against.** `.age` files in a git repo, first line = password, rest = metadata. The Passage project proves this format has community traction. **Design decision:** For single-identity MVP, try the configured identity against every `.age` file. No `.age-recipients` parsing needed. If decryption fails, return a safe error. Multi-identity + recipient-aware decryption is post-MVP.

4. **Local config + zeroize-per-decrypt for identity storage (revised from original premise).** Identity stored in app config for MVP with zeroize-per-decrypt-call in Rust core from day one. Android Keystore integration deferred to a later phase via the plugin interface. **Rationale:** Zeroize is ~20 lines of pure Rust with no Android dependency. Keystore adds ~1 day of platform debugging on top of an already ambitious build.

5. **Read-only is the right MVP scope.** Ship clone/list/search/decrypt/copy, get feedback, then expand.

## Cross-Model Perspective

Second opinion (Claude subagent) raised three key points:

1. **Identity handling is the attack surface.** If trust IS the product, the identity file can't be an afterthought. The zeroize-per-decrypt pattern was adopted from this feedback.

2. **Build as a Tauri plugin, not just an app.** Package the core as `tauri-plugin-age-store` so other Tauri apps can consume it. Become infrastructure. **Deferred to post-MVP** — the plugin interface should be clean enough to extract later, but adding plugin packaging ceremony before a working app is over-engineering.

3. **Reference `str4d/wage`.** A Vue-based age encryption/decryption web app using the same Rust crate (compiled to WASM). Same Vue stack, relevant UI patterns for key management and encrypt/decrypt flows.

## Approaches Considered

### Approach A: Core-first, mobile later (CHOSEN)

Build the Rust core (crypto + store parsing + git) on desktop first, prove it works with test fixtures, then wire into Tauri Android target. The Rust core is platform-agnostic and drops in unchanged.

**Effort:** M (total ~3 days across all phases) | **Risk:** Low | **Completeness:** 7/10

- ✅ Rust core tested on desktop in hours. Test fixtures with real `.age` files prove crypto works before touching Android toolchains.
- ✅ Platform-agnostic core. Android comes later as an engineering task, not a research task.
- ✅ Fastest path to answering "does the crypto even work?" — end of day 1.
- ❌ Delays mobile constraints (clipboard, screen size, back nav) until later.
- ❌ Less exciting demo — desktop app instead of the Android app you want to use.

### Approach B: Vertical slice on Android

Tauri v2 Android target from day one, building Rust core + Vue UI + mobile plugin interface in parallel.

**Effort:** M | **Risk:** Medium | **Completeness:** 9/10

- ✅ Building on the actual target device. Every decision shaped by real constraints.
- ✅ Most motivating — "I built a password manager for my phone this weekend."
- ❌ Android toolchain (SDK + NDK + JDK + Gradle + tauri android init) can eat half a day.
- ❌ Debugging Rust crypto issues on Android is much harder than desktop.

### Approach C: Plugin-first architecture

Build the Rust core as a reusable Tauri plugin (`tauri-plugin-age-store`), then consume it from the app.

**Effort:** M | **Risk:** Medium | **Completeness:** 8/10

- ✅ Reusable core. Other Tauri apps can consume age+gopass as a plugin.
- ✅ Forces clean API boundaries from day one.
- ❌ Plugin packaging ceremony before a working app. Risk of over-engineering.
- ❌ Building for imaginary consumers before proving it works for the one real user.

## Recommended Approach

**Approach A: Core-first, mobile later.**

Build sequence:

### Phase 1: Rust core library (desktop, ~1 day)

Plain Rust library crate. Depend on `age` (from `str4d/rage`) and `git2`. Four functions tested against a local test repo with real `.age` files:

- `clone_repo(url: &str, dest: &Path) -> Result<(), AppError>`
- `pull_repo(repo_path: &Path) -> Result<PullResult, AppError>` — fetch + ff-only merge
- `list_entries(repo_path: &Path) -> Result<Vec<Entry>, AppError>` — walkdir, skip `.git`, filter `.age`
- `decrypt_entry(repo_path: &Path, entry_path: &str) -> Result<DecryptedEntry, AppError>` — loads identity from secure storage internally, decrypts, parses first line as password, zeroizes identity after use

**Test fixtures:** A bare git repo with 2-3 `.age` files encrypted to a test key.

**Security in core from day one:**

- Identity zeroized per-decrypt-call (use `zeroize` crate on the identity bytes)
- `DecryptedEntry` fields use `Zeroizing<String>` wrapper — zeroized on Drop after IPC serialization. No `secrecy` crate: it blocks `Serialize` and fights Tauri IPC. A custom `Debug` impl (5 lines) prevents accidental log leakage instead.
- Error messages sanitized — no secrets, no identity content, no raw decrypted text
- Logging filtered — no secret content at any log level
- `DecryptedEntry` has a custom `Debug` impl that redacts all fields (`[REDACTED]`)

### Phase 2: Desktop Tauri app (~0.5 day)

`npx create tauri-app@latest` with Vue 3 + TypeScript + Vite. Wire Rust functions as `#[tauri::command]`.

Three-page Vue app:

1. **Setup page:** Git URL input, PAT input (for HTTPS auth), age identity input (paste or file import), clone repo
2. **Entry list page:** List all `.age` entries, case-insensitive substring search on name/path, pull-to-refresh (ff-only). Each entry shows a "Copy" button that calls `copy_password` (password never reaches WebView). Tap entry name to navigate to detail view.
3. **Entry detail page:** Calls `show_password` on demand. Password hidden by default, show/hide toggle. 30s auto-clear timer (nulls reactive refs). Display notes. On page leave (`onBeforeUnmount` + `onBeforeRouteLeave`), null out all reactive refs holding decrypted content. No `<KeepAlive>`, no `localStorage`.

### Phase 3: Android target (~1-2 days)

- `tauri android init` — set up Android toolchain
- Wire Tauri mobile plugin interface for Android:
  - `secure_save_identity` / `secure_load_identity` (mock: local file with app-private permissions for MVP, Android Keystore-backed encryption post-MVP)
  - Clipboard: use `tauri-plugin-clipboard-manager` (existing Tauri v2 plugin with Android support) for copy, with Rust-side `tokio::time::sleep(30s)` then overwrite clipboard for auto-clear. No custom Kotlin clipboard code needed.
  - `enable_flag_secure` (deferred — prevent screenshots)
- Test on real Android device
- Handle Android-specific: back navigation, screen size, clipboard behavior

### Phase 4: Polish & publish

- Android FLAG_SECURE
- Android Keystore for identity storage (replace mock)
- Publish APK via GitHub Release

### Future / Post-MVP

- Repo-as-trust-boundary visualization (diff view after pull showing changed/new/removed entries with commit metadata)
- Extract core as `tauri-plugin-age-store` if adoption warrants it

## Rust Command API

```rust
// Non-sensitive operations — no secret data involved
#[tauri::command]
fn clone_repo(url: String, dest: String) -> Result<(), AppError>

#[tauri::command]
fn pull_repo(repo_path: String) -> Result<PullResult, AppError>

#[tauri::command]
fn list_entries(repo_path: String) -> Result<Vec<Entry>, AppError>

// Primary sensitive operation — password never crosses IPC
// Decrypts, copies to clipboard, zeroizes — all on Rust side
#[tauri::command]
fn copy_password(repo_path: String, entry_path: String) -> Result<CopyResult, AppError>

// Secondary sensitive operation — password crosses IPC with strict lifecycle
// Only used when user needs to view the password
#[tauri::command]
fn show_password(repo_path: String, entry_path: String) -> Result<SensitiveContent, AppError>
```

**Security design:**

- All decrypt operations load the identity from secure storage internally — never passed from Vue.
- `copy_password` is the primary operation. Password never reaches WebView. Returns only `CopyResult` (no secret data).
- `show_password` is the secondary operation. Password crosses IPC as `SensitiveContent` with `Zeroizing<String>` fields. Vue component must follow the lifecycle protocol (30s auto-clear, null refs on page leave).
- After every decrypt operation, identity bytes and decrypted content are zeroized on the Rust side.

## Error Codes

All errors use safe messages — no secret content, no identity fragments, no raw decrypted text.

| Code               | Meaning                                                | Retryable                         |
| ------------------ | ------------------------------------------------------ | --------------------------------- |
| `CLONE_FAILED`     | Git clone failed (bad URL, auth, network)              | Yes                               |
| `PULL_FF_FAILED`   | Fast-forward pull failed (diverged branches)           | No — user must resolve on desktop |
| `DECRYPT_FAILED`   | Age decryption failed (wrong identity, corrupted file) | No                                |
| `INVALID_IDENTITY` | Identity is not valid `AGE-SECRET-KEY-...` format      | No                                |
| `NO_REPO`          | No repository found at configured path                 | No — re-run setup                 |
| `NO_IDENTITY`      | No identity configured                                 | No — re-run setup                 |
| `NETWORK_ERROR`    | Network unreachable                                    | Yes                               |
| `ENTRY_NOT_FOUND`  | Entry path does not exist in repo                      | No                                |
| `IO_ERROR`         | Filesystem error reading entry                         | Yes                               |

## Data Structures

```rust
struct Entry {
    path: String,     // relative path, used as key (e.g., "cloud/aws/root.age")
    name: String,     // display name (e.g., "aws/root")
}

// Internal only — never crosses IPC. Used by copy_password and show_password.
struct DecryptedEntry {
    password: Zeroizing<String>, // first line of decrypted content, zeroized on Drop
    notes: Zeroizing<String>,    // everything after first newline, zeroized on Drop
}

// Returned by copy_password — no secret data, safe to send to WebView
struct CopyResult {
    success: bool,
    entry_name: String,      // for toast: "Copied gmail/main"
    cleared_after_secs: u32, // user knows auto-clear timing
}

// Returned by show_password — contains secrets, strict Vue lifecycle required
struct SensitiveContent {
    password: Zeroizing<String>, // zeroized on Drop after IPC serialization
    notes: Zeroizing<String>,    // zeroized on Drop after IPC serialization
}

struct PullResult {
    changed: bool,    // whether any commits were pulled
    head: String,     // short hash of new HEAD
}

struct AppError {
    code: String,     // machine-readable error code (see Error Codes section)
    message: String,  // safe human-readable message (no secrets)
}
```

**Design decisions:**

- `Entry.id` removed — `path` serves as the unique key. No redundant field.
- `DecryptedEntry.raw` removed for MVP — YAGNI. The password + notes split covers the gopass/pass format. If structured metadata parsing is needed later, it can be added without breaking the API.
- `DecryptedEntry` fields use `Zeroizing<String>` wrappers (from `zeroize` crate — a wrapper that calls `zeroize()` on Drop). After the Tauri IPC serialization completes and the struct is dropped, the Rust-side strings are zeroized. WebView-side clearing is the Vue component's responsibility (null refs on page leave).
- No `secrecy` crate. `Secret<T>` intentionally blocks `Serialize`, which conflicts with Tauri IPC. A custom `Debug` impl prevents accidental log leakage with zero dependencies.

## Security Architecture

### Threat Model Overview

Sensitive data (age identity, decrypted passwords) flows through multiple runtime boundaries. Each boundary has different zeroize capabilities. This section documents every boundary, what controls it, and what it cannot do.

### Trust Boundary Map

```
┌─────────────── Android App Process ──────────────────────────┐
│                                                               │
│  ┌──────────────┐   JNI (in-process)  ┌──────────────────┐  │
│  │   Rust core   │◄═══════════════════►│  Kotlin (JNI)    │  │
│  │               │                     │  (plugin side)   │  │
│  └──────┬───────┘                     └────────┬─────────┘  │
│         │                                      │             │
│    Tauri IPC (JSON)                    Android APIs          │
│    ┌────┴─────┐                     ┌─────────┴──────────┐ │
│    │ WebView   │                     │ Keystore /          │ │
│    │ (JS heap) │                     │ ClipboardManager    │ │
│    └──────────┘                     └────────────────────┘ │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### Boundary 1: Rust Internal Memory

**Controllable: ✅**

`Zeroizing<String>` zeroizes on Drop. Identity bytes zeroized per-decrypt-call. The `DecryptedEntry` custom `Debug` impl returns `[REDACTED]` for all fields.

```rust
impl std::fmt::Debug for DecryptedEntry {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        f.debug_struct("DecryptedEntry")
            .field("password", &"[REDACTED]")
            .field("notes", &"[REDACTED]")
            .finish()
    }
}
```

### Boundary 2: Rust → WebView IPC

**Controllable: ⚠️ Partially — with architectural mitigation**

This is the highest-risk boundary. Once `DecryptedEntry` is serialized to JSON and sent to the WebView, copies exist in: JSON buffer (Rust side), IPC transport, JS heap (JSON.parse), Vue reactive state (Proxy), and DOM (even if hidden). JavaScript strings are immutable and GC-managed — setting a ref to `null` does not zero the old string. GC does not overwrite memory.

**Mitigation: split operations by sensitivity.** The primary use case (copy password) is redesigned to never cross this boundary.

#### Primary operation: `copy_password` — password never reaches WebView

```rust
#[tauri::command]
fn copy_password(repo_path: String, entry_path: String) -> Result<CopyResult, AppError> {
    // Identity loaded from secure storage, not from Vue
    let identity = secure_storage::load_identity()?;

    // Decrypt entirely on Rust side
    let entry = decrypt_and_parse(&repo_path, &entry_path, &identity)?;
    zeroize(identity);

    // Copy to clipboard (goes through Kotlin → Android ClipboardManager)
    clipboard::write(&entry.password)?;
    spawn_clipboard_clear(30); // tokio::spawn, 30s overwrite

    // Zeroize all decrypted content
    zeroize(entry.password);
    zeroize(entry.notes);

    // Return only metadata — no secret crosses IPC
    Ok(CopyResult {
        success: true,
        entry_name: entry.name,
        cleared_after_secs: 30,
    })
}
```

```rust
struct CopyResult {
    success: bool,
    entry_name: String,      // for toast: "Copied gmail/main"
    cleared_after_secs: u32, // user knows auto-clear timing
}
```

**Net effect:** The most common operation (90%+ of usage) has zero WebView exposure to the password. The attack surface for `copy_password` is: Rust memory (✅ zeroized) → Kotlin (✅ ByteArray zeroizable) → Android clipboard (✅ 30s overwrite). No JS heap involvement.

#### Secondary operation: `show_password` — password crosses IPC with strict lifecycle

Only for the rare case where the user needs to see the password. Password crosses IPC, but with enforced lifecycle:

```rust
#[tauri::command]
fn show_password(repo_path: String, entry_path: String) -> Result<SensitiveContent, AppError> {
    let identity = secure_storage::load_identity()?;
    let entry = decrypt_and_parse(&repo_path, &entry_path, &identity)?;
    zeroize(identity);
    // Zeroizing<String> zeroizes on Drop after IPC serialization
    Ok(SensitiveContent {
        password: Zeroizing::new(entry.password),
        notes: Zeroizing::new(entry.notes),
    })
}
```

Vue component protocol for `show_password`:

```typescript
// PasswordDetail.vue — strict lifecycle management
const password = ref<string | null>(null);
const notes = ref<string | null>(null);
let autoHideTimer: ReturnType<typeof setTimeout> | null = null;

async function onEntryClick(entryPath: string) {
  const result = await invoke<SensitiveContent>("show_password", { entryPath });
  password.value = result.password;
  notes.value = result.notes;

  // Auto-clear after 30 seconds
  autoHideTimer = setTimeout(() => {
    password.value = null;
    notes.value = null;
  }, 30_000);
}

onBeforeUnmount(() => {
  password.value = null;
  notes.value = null;
  if (autoHideTimer) clearTimeout(autoHideTimer);
});

onBeforeRouteLeave(() => {
  password.value = null;
  notes.value = null;
  if (autoHideTimer) clearTimeout(autoHideTimer);
});
```

**Vue-side rules (must not violate):**

- ❌ No `localStorage` / `sessionStorage`
- ❌ No `<KeepAlive>` on this component
- ❌ No passing password as prop to child components (unless they follow the same protocol)
- ❌ No `console.log` with secret content

**Acknowledged limitation:** Even with all mitigations, JS heap may retain stale copies until GC collects them. GC does not zero memory. This is a WebView runtime limitation, not a fixable bug. Documented here as a known trust boundary.

### Boundary 3: Rust → Kotlin (JNI Plugin Bridge)

**Controllable: ✅ with discipline — better than WebView**

Rust and Kotlin run in the same Android process. JNI is in-process FFI (not network). Data is serialized through Tauri's plugin bridge. Two data paths:

#### Path 1: Identity (Kotlin → Rust)

Post-MVP when Android Keystore is integrated:

```
Kotlin: Keystore decrypt → ByteArray → Tauri plugin bridge → Rust
```

**Rule: always use `ByteArray`, never `String`.** JVM `String` is immutable — once created, it cannot be zeroed and lingers until GC. `ByteArray` is mutable and can be zeroed in-place:

```kotlin
class SecureStoragePlugin : Plugin() {
    @Command
    fun loadIdentity(invoker: Invoker) {
        val identity: ByteArray = keyStoreDecrypt(ENCRYPTED_IDENTITY_KEY)
        invoker.resolve(identity)
        identity.fill(0) // In-place zero — works because ByteArray is mutable
    }
}
```

#### Path 2: Password (Rust → Kotlin → Android Clipboard)

```
Rust: decrypt → Tauri clipboard plugin bridge → Kotlin → Android ClipboardManager
```

**MVP:** Use `tauri-plugin-clipboard-manager`. It likely uses `String` internally on the Kotlin side, meaning JVM heap retains an immutable copy until GC. Acceptable for MVP because:

- Same process, not cross-process (except clipboard service by design)
- Android process kill clears all JVM memory
- No root-level attacker can read another process's memory
- 30s clipboard overwrite closes the most exposed window

**Post-MVP upgrade:** Custom clipboard plugin using `ByteArray` throughout:

```kotlin
class SecureClipboardPlugin : Plugin() {
    @Command
    fun copyWithTtl(invoker: Invoker, content: ByteArray, ttlSeconds: Int) {
        val clipboard = getSystemService(Context.CLIPBOARD_SERVICE) as ClipboardManager
        val clip = ClipData.newPlainText("password", String(content, Charsets.UTF_8))
        clipboard.setPrimaryClip(clip)

        content.fill(0) // Zero ByteArray immediately

        Handler(Looper.getMainLooper()).postDelayed({
            val clearClip = ClipData.newPlainText("cleared", "")
            clipboard.setPrimaryClip(clearClip)
        }, ttlSeconds * 1000L)

        invoker.resolve(null)
    }
}
```

### Boundary 4: Tauri Plugin Bridge Serialization

**Controllable: ⚠️ Trust dependency on Tauri internals**

The Tauri plugin bridge serializes data between Rust and Kotlin. The serialization buffer lifecycle is controlled by Tauri, not by this app. We cannot guarantee the bridge zeroizes its internal buffers after use.

**Mitigation:** Document this as a trust boundary. The data that transits this bridge (identity bytes, password content) should have the shortest possible exposure. The `copy_password` pattern (Rust-side decrypt → immediate copy → immediate zeroize) minimizes the time any secret exists in this buffer.

### Boundary 5: Android System Clipboard

**Controllable: ✅ By design**

By design, the password reaches the Android clipboard — that's what "copy" means. Android 10+ restricts background app clipboard access. The 30-second overwrite ensures the exposure window is bounded. `ClipData` labels should use generic text ("password"), not the actual entry name.

### Trust Boundary Summary

| #   | Boundary                   | Data                                      | Zeroizable?            | Risk   | Mitigation                                                                       |
| --- | -------------------------- | ----------------------------------------- | ---------------------- | ------ | -------------------------------------------------------------------------------- |
| 1   | Rust internal memory       | Identity, password                        | ✅ `Zeroizing<String>` | Low    | Drop auto-zeroize + custom Debug                                                 |
| 2   | Rust → WebView IPC         | Password (show only)                      | ❌ JS immutable + GC   | Medium | `copy_password` avoids IPC; `show_password` has 30s auto-clear + page-leave null |
| 3   | Rust → Kotlin (JNI)        | Identity (Keystore), password (clipboard) | ✅ `ByteArray.fill(0)` | Low    | Always ByteArray, never String                                                   |
| 4   | Tauri plugin bridge buffer | All transiting data                       | ⚠️ Tauri internals     | Low    | Minimize time-in-buffer; trust dependency documented                             |
| 5   | JVM heap (String)          | Password via clipboard plugin             | ❌ JVM immutable       | Medium | MVP: accept. Post-MVP: custom ByteArray plugin                                   |
| 6   | WebView JS heap            | Password (show only)                      | ❌ GC, no overwrite    | Medium | 30s auto-clear + page-leave null + no KeepAlive. Acknowledged limitation         |
| 7   | Android clipboard          | Password                                  | ✅ 30s overwrite       | Low    | `ClipData` label is generic, not entry name                                      |

**Architectural principle:** The primary operation (copy) touches boundaries 1→3→4→7 (all zeroizable or by-design). The secondary operation (show) touches boundaries 1→2→6 (partially controllable, acknowledged limitation). This is the best achievable with current WebView and JVM runtimes.

## Android Plugin Interface

```rust
// Tauri mobile plugin — Kotlin side implements, Rust side calls
trait SecureStorage {
    fn secure_save_identity(identity: &[u8]) -> Result<(), StorageError>;
    fn secure_load_identity() -> Result<Vec<u8>, StorageError>;
    fn secure_delete_identity() -> Result<(), StorageError>;
}

trait DeviceSecurity {
    fn enable_flag_secure() -> Result<(), SecurityError>;
}
```

**MVP implementations:**

- `SecureStorage`: Identity written to app-private file with restrictive permissions. No Keystore encryption. **Mock quality:** the file is app-private on Android (other apps can't read it), but not encrypted-at-rest. Acceptable for single-user MVP on a device you control.
- `DeviceSecurity`: No-op for MVP. `FLAG_SECURE` added post-MVP.

**Post-MVP Keystore upgrade:** When `SecureStorage` is upgraded to Android Keystore, the identity crosses the Kotlin→Rust boundary. **Kotlin-side rule: always use `ByteArray`, never `String`.** JVM `String` is immutable and cannot be zeroed. `ByteArray` is mutable and can be zeroed in-place with `.fill(0)`. See Security Architecture section for the full Kotlin code pattern.

**Clipboard — MVP:** Uses `tauri-plugin-clipboard-manager` (existing Tauri v2 plugin with Android/iOS support) for copy. Auto-clear implemented in Rust via `tokio::time::sleep(Duration::from_secs(30))` followed by overwriting clipboard with empty string. The plugin may use `String` internally on the Kotlin side — JVM heap retains an immutable copy until GC. Acceptable for MVP (same process, no root attacker). No custom Kotlin clipboard code needed.

**Clipboard — Post-MVP upgrade:** Custom `SecureClipboardPlugin` using `ByteArray` throughout. Password received as `ByteArray`, passed to Android ClipboardManager, then `.fill(0)` immediately. Eliminates JVM `String` retention. See Security Architecture section for the full Kotlin code pattern.
